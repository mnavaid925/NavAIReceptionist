## Feature catalog (this sub-module only)

### Own Profile
- **Self-service name/phone edit** — first/last/full name + primary phone editable by the signed-in user for themselves · seen in: universal (every platform separates "my profile" from "manage other users": Dialpad, Retell, Vapi, PolyAI) · priority: table-stakes · model: reuses `accounts.User` (`request.user`, tenant-scoped, no query needed) · realtime: post-call (pure web CRUD, not call-runtime) · tool-surface: pure UI — `TenantModelForm`-based form with `Meta.fields = {first_name, last_name, full_name, primary_phone}` only · buildable now.
- **No self-service tier/status/`is_provider` change** — the own-profile form must never expose `tier`, `status` or `is_provider` · seen in: every RBAC platform surveyed separates "edit my profile" from "edit my own permissions" (Retell/Vapi/PolyAI/Dialpad all gate role changes behind an elevated actor) · priority: **REQUIRED** (a self-editable tier field is a privilege-escalation bug, the same class the project's Vulnerability rules call out) · model: reuses `accounts.User` (field-level allowlist in the profile form's `Meta.fields`) · realtime: post-call · tool-surface: pure UI/backend (form field restriction) · buildable now.
- **Full-name auto-derivation, surfaced not re-implemented** — `full_name` is already computed from first+last when left blank (`User.clean()`/`.save()`) · seen in: general pattern (Dialpad/Retell show a computed display name) · priority: common · model: reuses `accounts.User.full_name` (verified, model already does this) · realtime: post-call · tool-surface: pure UI (help text only — no new logic) · buildable now.

### Deactivation Instead of Deletion
- **The mandatory delete route performs a soft deactivation, never `.delete()`** — the `user_delete` POST route required by the project's CRUD Completeness Rules sets `status='inactive'` and redirects to the list · seen in: Dialpad (even its "delete" is a 72h-restorable soft state, not immediate hard delete), general SaaS "deactivate don't delete" pattern for any record with historical FK references · priority: **REQUIRED** — this is exactly how the mandatory delete route (CRUD Completeness Rules) reconciles with the sub-module's own explicit "Deactivation Instead of Deletion" bullet; an actual `.delete()` would break referential integrity the moment `scheduling.Appointment` exists · model: reuses `accounts.User.status` (verified) — no new model, no new field, just the handler body · realtime: post-call · tool-surface: pure backend · buildable now.
- **Self-deactivation guard** — a user, even an owner, cannot deactivate their own account through this route · seen in: general SaaS pattern (block "remove yourself"); Dialpad's removal flow explicitly reviews "admin coverage" before confirming, the same guard class · priority: table-stakes · model: reuses `accounts.User` (`pk == request.user.pk` check in the view) · realtime: post-call · tool-surface: pure backend · buildable now.
- **Last-owner guard** — block deactivating the tenant's last remaining active owner-tier user, so a business can never lock itself out of user management · seen in: general pattern, analogous to Dialpad's "admin coverage" warning before removal · priority: common · model: reuses `accounts.User` (tenant-scoped count query: `tier=owner, status=active`, excluding the target) · realtime: post-call · tool-surface: pure backend · buildable now.

- **Reactivation via the edit form** — no separate "reactivate" button; see "Status assignment at create/edit" above, restated here to close the loop with this bullet group · priority: table-stakes · model: reuses `accounts.User.status` · realtime: post-call · tool-surface: pure UI/backend · buildable now.

### Beyond the bullets
- **GDPR rectification (Art. 16) satisfied by Own Profile** — a user's right to correct their own personal data (name, phone) is met by the self-service form; no separate "request a correction" workflow is needed · priority: **REQUIRED** (`accounts.User` carries personal data: email, phone, name) · model: reuses `accounts.User` · realtime: post-call · tool-surface: pure UI · design note, not a new feature.
- **Deactivation as the documented data-retention rationale for staff records** — because deletion is never offered, the product's retention documentation should record that staff/provider accounts are retained (deactivated, never erased) specifically to preserve referential integrity for historical appointments, a deliberate trade-off against literal "right to erasure" for employee-type records, common under legitimate-interest processing of HR/staff data · priority: **REQUIRED** (GDPR retention rationale needs to exist somewhere, even though it is a documentation item, not code) · model: none · N/A — write into the module skill file.
- **Team-member vs. bookable-directory-contact distinction (Goodcall)** — Goodcall separates a "team member" (dashboard access + agent-edit rights) from a "directory contact" (call-transfer/booking target, no login at all) · seen in: Goodcall · priority: informational only — this product's `accounts.User` deliberately conflates the two: a `tier=staff, is_provider=True` user both logs in and is bookable, a correct simplification given the eleven-model ceiling has no room for a second, non-login "bookable contact" entity distinct from `User` · model: none — no change recommended · realtime: post-call · tool-surface: none.

## Compliance & provider constraints

- **REQUIRED — role-gated write access.** A staff-tier user must never reach the create/edit/delete user views, nor set their own `tier`/`status`/`is_provider` via the own-profile form. This is 0.3's core security guardrail (privilege escalation), grounded in every RBAC platform surveyed gating "manage other users" behind an elevated role.
- **REQUIRED — GDPR rectification (Art. 16).** Satisfied by Own Profile; no gap to close beyond shipping that form.
- **REQUIRED — GDPR retention rationale.** Deactivation-not-deletion for staff/provider rows must be documented as a deliberate retention policy (legitimate interest: referential integrity for historical appointments), since erasure is never offered for these rows.
- **PII-logging discipline.** `accounts.User` carries email, phone and full name; never log a create/edit form's raw POST body at INFO — consistent with the project-wide PII-logging rule that already applies from 0.1 forward.
- **No telephony/recording/two-party-consent obligation.** 0.3 never touches `calls.CallSession` or a live call; the project's REQUIRED recording-consent items do not apply here.
- **No provider cost line.** 0.3 makes no Twilio/STT/TTS/LLM call, so nothing appends to `calls.CallSession.usage`. The only external dependency is outbound email for the invite-to-set-password flow — reuses 0.1's already-configured `EMAIL_BACKEND` (console in dev), no new integration.

## Recommended build scope (this pass)

This is a **CRUD sub-module over an already-existing model.** `accounts.User` was created in **0.1** to satisfy `AUTH_USER_MODEL`'s hard requirement of existing before the first `makemigrations` — 0.3 adds **ZERO new models and ZERO migrations**. Every field the researched features touch already exists on `accounts.User` / `accounts.UserLocation`. This is the deliberate exception to the "1–3 models" framing for a CRUD pass: the right model count when the CRUD target is a pre-existing foundation model is zero new tables, all new views/forms/templates.

- **`accounts.User`** (existing, verified — not created by 0.3) — the Create/Edit form's `Meta.fields = {email, username, first_name, last_name, full_name, primary_phone, tier, status, is_provider}`, justified by **User Create & Edit** + **Deactivation Instead of Deletion**. `tenant` is stamped server-side via `TenantModelForm` (never rendered, never a form field). `password` is **not** in `Meta.fields` — set via `set_unusable_password()` + an invite email on create, changed only through 0.2's dedicated change-password flow thereafter. `is_staff` (Django admin flag) and `provider_hours` (1.4's field) are explicitly excluded. The **Own Profile** form is a separate, narrower `ModelForm` over the same model: `Meta.fields = {first_name, last_name, full_name, primary_phone}` only.
- **`accounts.UserLocation`** (existing, verified — not created or written by 0.3) — **read-only** in this pass, via `user.assigned_locations()`, to render the "assigned locations" display on list/detail. Creating/editing `UserLocation` rows is **1.3's** job.
- **No third model.** None of the researched features justify one; the invite-to-set-password flow deliberately reuses 0.1's password-reset token mechanism precisely to avoid needing an `Invitation` table.

**New non-model surface this pass must add:** a `tier_required(...)` gate in `apps/accounts/views/_helpers.py` (shared by more than one entity-adjacent view: `user_list`/`_create`/`_detail`/`_edit`/`_delete`), since no such gate exists in the repo yet.

## Out of scope for this product (outside the seven capabilities)

- **Enterprise SSO-driven auto-role-assignment** — PolyAI explicitly does *not* auto-assign roles from SSO either, confirming even the enterprise platforms treat this as manual; this product has no SSO at all (already deferred in 0.1's own research), so doubly out of scope here.
- **Per-navigation-area granular RBAC** (PolyAI's "set permissions on individual items") — this product's 3-tier (owner/manager/staff) *is* the whole permission model; a finer-grained ACL would need schema growth beyond the eleven-model ceiling.
- **Directory contacts with no login** (Goodcall's second person-type) — this product has exactly one identity table for humans-with-accounts (`accounts.User`, plus `scheduling.Contact` for callers — a *third*, non-login "transferable staff contact" type is out of scope); Module 2's Transfer Settings (2.3) stores transfer destination numbers directly on `AgentSetting`, not as directory-contact rows.
- **Mobile app team management** (Rosie) — this product is a web (Tailwind/HTMX) application only.
- **Restore-window / soft-delete recovery UI** (Dialpad's 72h undo) — this product's "deactivate" already *is* the soft-delete floor; there is no harder delete state underneath it to recover from.
