3. **Rosie** — AI answering service for local/home-services businesses; the documented pattern is **one account per location**, with multi-location businesses either running parallel accounts or forwarding several numbers into one — no in-product location switcher — [Rosie plans](https://heyrosie.com/pricing)
4. **Goodcall** — AI phone agent purpose-built for franchise/multi-site operators; a dedicated multi-location management dashboard lets an operator configure a distinct agent (greeting, hours, routing) per site from one login — [Goodcall enterprise](https://www.goodcall.com/), franchise multi-location coverage confirmed via [ilounge comparison](https://www.ilounge.com/articles/top-ai-receptionist-for-companies-with-multiple-locations)
5. **Dialpad AI** — enterprise business-phone platform; "Centralized Office Management" gives an admin a single dashboard to view/configure/switch between offices instead of navigating each office's settings individually — [Add and Manage Multiple Offices](https://help.dialpad.com/docs/add-and-manage-multiple-offices), [Centralized Office Management](https://help.dialpad.com/docs/centralized-office-management)
6. **Retell AI / Vapi / Synthflow / Bland AI / PolyAI** (the mandated agent-platform competitor set) — **notable negative finding**: none of these publish a location/site switcher at all. They are single-workspace-per-login dashboards; a "location" is just another phone number/agent config with no unifying switcher, and agency/multi-client management is a documented gap ("no real multi-tenant dashboard... managing multiple brands or clients means juggling accounts manually") — [Retell AI blog: Vapi vs Synthflow](https://www.retellai.com/blog/vapi-vs-synthflow), general competitor surveys confirm the same absence for Bland AI and PolyAI. This confirms the switcher pattern for this product must be sourced from the receptionist-service and multi-location-retail competitors below, not the agent-builder platforms.
7. **Square Dashboard** (multi-location retail/appointments, cited per the task's landscape map) — POS/mobile app "Switch Location" action under a `More` menu; **as of the April 2026 release, per-team-member location permissions are enforced on writes** ("team members can only edit items... for the locations they have access to") — the exact isolation risk this sub-module's "Assignment Validation" bullet exists to prevent — [Manage multiple locations](https://squareup.com/help/us/en/article/5580-manage-multiple-locations-with-square), [April 2026 release notes](https://squareup.com/us/en/release-notes/release-notes-april-23rd)
8. **Toast POS** (multi-location restaurant management, cited per the task's landscape map) — location name in the top-left corner of every page is itself the switcher trigger (dropdown); a **separate, explicit "all locations" aggregate view** exists only in the reporting product (Toast Now), never in the operational/settings surface, where you instead pick a subset via "Select locations to manage" — [Switch Between Restaurant Locations](https://support.toasttab.com/en/article/Switching-Between-Toast-Restaurants), [Toast Now](https://pos.toasttab.com/products/toast-now)
9. **Mindbody** (multi-location wellness/fitness scheduling, cited per the task's landscape map) — the switcher is **gated by a per-staff permission** ("Allow users to switch between locations while logged in"); if unchecked, that staff member is pinned to exactly one location and must log out/back in to work elsewhere — the strongest documented analogue to "assignment validation" as a hard gate rather than a convenience — [Multi-location management guide](https://support.mindbodyonline.com/s/article/How-do-I-switch-locations-Frederick?language=en_US), [Staff Permissions](https://support.mindbodyonline.com/s/article/Staff-Permissions-Settings?language=en_US)

## Feature catalog (this sub-module only)

### Assigned-Location List
- **Read-only "locations you can work in" list** — a set derived purely from assignment rows, never hand-typed · seen in: Mindbody (staff permission scopes an explicit list), Goodcall (multi-location dashboard lists the operator's sites), Square (location list under Account & Settings) · priority: table-stakes · model: reuses `accounts.UserLocation` via `User.assigned_locations()` — **already implemented**, tenant-scoped (not location-scoped, by design — it's the table that defines which locations are reachable) · realtime: post-call / N/A — pure web-session concern, not the call runtime · tool-surface: pure UI, no LLM tool · buildable now, zero new code beyond a template that lists `user_locations` (already in context).
- **Empty-set handling (a user assigned to zero locations)** — the UI must say so plainly rather than silently blanking every location-scoped page · seen in: general SaaS practice; the closest documented analogue is Mindbody's permission gate producing a single-location lock rather than a blank state · priority: **REQUIRED** (a user with zero rows must get an actionable message, not a confusing empty dashboard — this is the corner the middleware already returns `None` for, but nothing today explains why) · model: reuses `accounts.UserLocation` (empty queryset) · realtime: N/A · tool-surface: pure UI (a banner/empty-state component) · buildable now.

### Session Active Location
- **One active location per session, exposed as `request.location`** — **already fully built** by `ActiveLocationMiddleware`; 0.4 adds no backend behaviour here, only the interactive control that writes to it · seen in: Toast (top-left location name persists per session until changed), Square (device/session-level "current location") · priority: table-stakes · model: no model — session key `active_location_id`; the *authorization* re-check is `accounts.UserLocation` · realtime: N/A · tool-surface: pure backend/session · **already buildable/built**.
- **Auto-activate the sole assignment, no prompt** — a single-location user is never shown a switcher at all (the `_topbar.html` gate already does this: no `<select>` when `user_locations` has one entry, because with exactly one assignment the middleware has already set it and there's nothing to choose) · seen in: universal pattern across every multi-location competitor surveyed (none of them force a choice when there is only one option) · priority: table-stakes · model: reuses existing middleware/helpers, no change · realtime: N/A · tool-surface: pure UI (topbar conditional, already correct) · already built.
- **Remember the last-used location across logins (persistent default), WITHOUT a new column** — several competitors (Toast, Square) keep the last-selected location "sticky" per device/session rather than re-prompting at every login · priority: **common**, genuinely useful, but explicitly constrained by this pass's zero-migration rule · model: **no new field** — deliver this by **not clearing `active_location_id` from the session at logout**... except `SessionPolicyMiddleware`'s docstring is explicit that logout flushes the whole session and a re-login re-derives from `UserLocation` "rather than restoring a stale choice," and `django.contrib.auth.logout()` rotates/flushes the session by default. The zero-model-safe way to approximate "remembered default" **without** a `UserLocation.is_default` column is a **signed cookie** (`django.core.signing.dumps`/`loads`, `max_age`) set alongside the session write in `set_active_location()`, read back by the login view/`activate_sole_location` path to pre-select (not silently activate) a location on next login — still re-validated against `assigned_locations()` before it is trusted, exactly like the session id is today. This is additive to the existing helpers, not a schema change · realtime: N/A · tool-surface: pure backend (cookie + existing validation path) · buildable now, **recommend as a should-have, not a must-have for this pass** — the documented bullets don't ask for cross-session persistence, only "forces a choice when the user's first assignment loads," which is already satisfiable without it.

### Assignment Validation
- **Server-side re-validation on every switch attempt (not just at login)** — **already built**: `ActiveLocationMiddleware._resolve()` re-checks the stored id against `assigned_locations()` on every request, and `_helpers.set_active_location()`'s docstring places the burden of proof on the caller. 0.4's job is to write the **view** that is that caller: `POST accounts:switch_location` must re-derive the requested location from `request.user.assigned_locations().filter(pk=posted_id).first()` — **never** `Location.objects.get(pk=posted_id)` — and silently ignore/reject (with a message) a `pk` that doesn't resolve, exactly mirroring the topbar `<select>`'s own options · seen in: Mindbody (permission-gated switch, hard denial for unassigned staff), Square (April 2026 write-permission enforcement is the same shape one layer down — writes respect assigned locations even when the switcher itself is open) · priority: **REQUIRED** — this is the sub-module's own stated purpose ("the boundary a cross-location IDOR would cross") · model: reuses `accounts.UserLocation` (query only, no write) · realtime: N/A — HTTP request/response, not the call runtime · tool-surface: pure backend view; **not** an LLM tool — a caller-facing voice session never switches a staff member's active location · buildable now.
- **POST-only switch, CSRF-protected, redirect-back** — matches the markup already shipped in `_topbar.html` (`<form method="post">`, `{% csrf_token %}`) · seen in: universal SaaS practice (Toast/Square switchers are themselves POST/session-mutating actions even when styled as a dropdown) · priority: table-stakes · model: none · realtime: N/A · tool-surface: pure backend view, redirects to `request.META.get('HTTP_REFERER')` or a safe default (dashboard) — never trusts an open redirect target from the form itself · buildable now.
- **Junk/foreign pk degrades to "no change," never a 500 or a silent wrong-location activation** — mirrors the project-wide filter rule ("a junk value must degrade to no filter, never raise") applied to the switcher · seen in: this product's own `CLAUDE.md` filter rules, generalized; no competitor documents their failure mode publicly, so this is a first-party requirement, not a market-copied one · priority: **REQUIRED** (it is the literal cross-location IDOR the sub-module exists to close) · model: none · realtime: N/A · tool-surface: pure backend · buildable now.

### Location Context Header
- **Active location always visible in the shell** — **already built**: `_topbar.html`'s `.location-switcher` (interactive) / read-only `<span>` (fallback) both show `active_location.name` with a map-pin icon whenever `active_location` is set · seen in: Toast (location name top-left on every page), Dialpad (office name in the admin chrome), Goodcall (per-site label in the multi-location dashboard) · priority: table-stakes · model: reuses `tenants.Location` (read) · realtime: N/A · tool-surface: pure UI · already built — 0.4 need only stop suppressing the `<select>` branch by shipping the url.
- **Forced choice when no active location and 2+ assignments** — **the one genuinely open gap.** Today `active_location` is simply `None` for such a user and nothing tells them to pick one; the topbar even hides the switcher UI in that state today because `{% if active_location %}` wraps the whole block in `_topbar.html`, so a 2+-assignment user with nothing chosen currently sees **no switcher at all**, only silence. 0.4 must both (a) show the switcher even when `active_location` is `None` (loosen the topbar's outer guard to `{% if user_locations %}` and render the `<select>` with no option pre-selected / a "Choose a location…" placeholder), and (b) redirect a first-touch dashboard/module view to a small "choose your location" interstitial instead of rendering a page whose every location-scoped query is empty · seen in: Slack/GitHub-style forced-org-choice on first login is the closest documented analogue in general B2B SaaS (not one of the ten fixed competitors, cited only as the general UX pattern name); among the fixed competitors, Mindbody's "must log out/in to switch" is the closest hard-gate precedent, and Goodcall's dashboard requires picking a site before showing that site's call data · priority: **REQUIRED per the sub-module's own bullet** ("forces a choice when the user's first assignment loads") · model: none — pure view/template logic keyed off `request.location is None and len(user_locations) != 1` · realtime: N/A · tool-surface: pure UI/view · buildable now.
- **Per-page confirmation that an action is scoped to the shown location** — e.g., "Booking created at **Downtown**" rather than a bare success toast, so a user switching frequently never mis-attributes an action to the wrong site · seen in: general multi-location SaaS practice (Toast's per-page location name serves this implicitly); not separately documented by name at any of the ten competitors · priority: common · model: none — a template convention (include `{{ active_location.name }}` in success `messages.*` calls in later sub-modules) · realtime: N/A · tool-surface: pure UI convention · **note for `todo`**: this is a cross-cutting UI convention future sub-modules should follow, not a page 0.4 itself ships — record it, don't build a page for it here.

### Beyond the bullets
- **"All locations" aggregate view for an owner** — Toast Now and Square's owner-level reporting both offer a read-only rollup across every site; **both draw a hard line that the rollup is read-only/reporting-only** — no write action in either product is ever performed against "all locations" at once; Square's own April 2026 change tightened this further by making even *item edits* respect per-location assignment. Toast's operational/settings surface (as opposed to its reporting product) still requires picking a specific location or an explicit subset, never a blanket "all" · priority: differentiator, and a **security-relevant one** — an "all locations" *write* context would let a user act on a location they have no `UserLocation` row for, which is exactly the cross-location IDOR this sub-module exists to prevent · model: **do not build for this pass.** If ever added, it must be: (1) an owner-tier-only, **read-only** aggregation across the locations already in `assigned_locations()` (never a bypass of the assignment check), and (2) `request.location` itself must **never** become "all" — every location-scoped write still requires one concrete, assigned `location` id. Concretely this would live as a query-time UNION across `assigned_locations()` in a *reporting* view of a later module, not as a value `ActiveLocationMiddleware` can set · realtime: N/A · tool-surface: none · **deferred, and flagged as a risk if ever revisited** — see Compliance section.
- **Location-scoped notification/badge count in the switcher** (e.g. "3 live calls" per location, as Dialpad's centralized office view shows per-office activity) · seen in: Dialpad (Centralized Office Management dashboard shows per-office status) · priority: differentiator · model: would read `calls.CallSession` (Module 5's model, not yet built) filtered by each assigned location — **out of scope for 0.4**, belongs with whichever module surfaces live-call counts (the topbar's existing `live_call_count` variable is already a hook for this, scoped to the *active* location only) · realtime: **live-call hot path** if ever built as a true live badge (would need a Channels push), but a simple per-request count is post-call/read-only · tool-surface: pure UI · deferred — no model to build in 0.4, and the underlying `CallSession` table doesn't exist yet.

## Compliance & provider constraints

- **No recording-consent, two-party-consent, HIPAA or GDPR-retention obligation is triggered by this sub-module.** 0.4 touches no `calls.CallSession`, no transcript, no recording and no PII collection beyond what `accounts.User`/`accounts.UserLocation` already store. It is a pure session/authorization surface.
- **The one REQUIRED item is access-control correctness, not a legal/compliance basis**: the switch view must re-derive the target location from `user.assigned_locations()` and never trust a posted `location` id directly (Multi-Tenancy & Location Rule #2, and Invariant-adjacent — this sub-module is *itself* the enforcement boundary the project's cross-location IDOR rule names explicitly). This is marked REQUIRED because it is the sub-module's stated reason for existing ("the boundary a cross-location IDOR would cross"), not because of an external law.
- **No provider cost line.** 0.4 calls no Twilio/STT/TTS/LLM API and appends nothing to `calls.CallSession.usage`. Its only "external" surface is the Django session/cookie mechanism already in use.
- **"All locations" write access is the one compliance-adjacent risk surfaced by this research** — see Beyond-the-bullets above: if a future pass ever introduces an aggregate view, it must stay strictly read-only and must never let `request.location` resolve to a non-single value, or every downstream `location=request.location` filter in the codebase silently becomes cross-location.

## Recommended build scope (this pass)

**This is a CRUD-adjacent sub-module that introduces ZERO new models and ZERO migrations**, per the hard constraint — it is scoped entirely against the existing `accounts.UserLocation` table and existing `ActiveLocationMiddleware`/`_helpers.py` machinery. There is no model recommendation here; what's missing is the **view + template surface**:

- **`accounts:switch_location`** — `POST`-only view. Reads `location` from `request.POST`, resolves it as `request.user.assigned_locations().filter(pk=posted_id).first()`, calls the already-built `set_active_location(request, location)` on success, silently no-ops (with an optional `messages.error`) on a miss/junk value, and redirects back (`HTTP_REFERER` validated to be same-origin, else a safe default like `accounts:dashboard`). No new model — reads `accounts.UserLocation` via `User.assigned_locations()`, writes only the session.
- **Loosen `_topbar.html`'s outer `{% if active_location %}` to `{% if user_locations %}`**, and render the `<select>` with a "Choose a location…" placeholder option when `active_location` is `None`, so a 2+-assignment user actually sees the control that lets them satisfy "forces a choice" instead of seeing nothing.
- **A small "choose your location" interstitial/banner** for the dashboard (and any other page reachable with `request.location is None`) — not a new model, a template partial that renders when `user.is_authenticated and not request.location and user_locations`, explaining why the page is empty and linking into the switcher.
- **Optional, lower-priority (recommend as should-have): a signed-cookie "last used location" hint** — read-only *suggestion* consumed at login/first-touch, always re-validated against `assigned_locations()` before being trusted, never a database column. Flag this to `todo` as optional; the four documented bullets are satisfiable without it.

## Out of scope for this product (outside the seven capabilities)

- **Partner/reseller cross-account switching** (Smith.ai's Partner Portal lets a partner sign into any managed client's account) — this product has no reseller/partner concept; a tenant is one business, and the switcher moves between that business's *own* locations only.
- **White-label multi-client agency dashboard** (the documented gap in Retell/Vapi/Synthflow, and the exact feature agencies ask those platforms for) — this product is single-tenant-per-login by design; there is no "manage many tenants from one session" surface anywhere in the seven capabilities.
- **Cross-regional data-sharing / "datashare" business grouping** (Mindbody's cross-regional setup, which links otherwise-separate businesses) — this product's tenant boundary is fixed; a location is always a site *within* one tenant, never a link across tenants.
- **Native mobile "switch location" app action** (Square's mobile app, Rosie's app) — the product is a web (Tailwind/HTMX) application; no mobile client is in scope.

## Deferred (later passes / integrations)

- **Signed-cookie "remembered last location" pre-select on login** — additive, zero-schema, but not required by the four documented bullets; build only if `todo`/product wants the convenience.
- **Read-only "all locations" aggregate/reporting view for an owner** — genuinely useful (Toast Now, Square reporting both have it) but carries real cross-location-leak risk if built carelessly; deferred until a reporting-focused module explicitly scopes it, and even then it must stay read-only and never let `request.location` itself become non-singular.
- **Per-location live-call badge in the switcher** — depends on `calls.CallSession` (Module 5, not built) and, if made truly live, on a Channels push; deferred entirely.
- **Per-page "scoped to <Location>" message convention** — a documentation/convention note for future sub-modules' `messages.*` calls, not a page 0.4 builds itself.
- **Admin-forced/default location assignment per staff member** (Mindbody's permission-gated "may this user switch at all") — the current model already achieves the tightest version of this for free: a user with exactly one `UserLocation` row never sees a switcher and can never reach another location, because `assigned_locations()` — not a separate permission flag — is the only source of truth. A separate "can switch" boolean would duplicate what having 1 vs 2+ assignment rows already expresses; not recommended even later unless a concrete gap appears.
