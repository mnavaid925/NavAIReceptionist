## Leaders surveyed (with source links)

1. **Dialpad AI** — enterprise business-phone/AI-receptionist platform; the most mature credential-management posture of the set — requires current password to change password, has a distinct "identity verification" surface for high-risk changes — [Change or Reset Your Password](https://help.dialpad.com/docs/change-or-reset-your-password), [Change an Email Address](https://help.dialpad.com/docs/change-an-email-address), [Account Security](https://help.dialpad.com/docs/account-security), [Identity Verification](https://help.dialpad.com/docs/identity-verification)
2. **Retell AI** — AI voice-agent platform; password reset is self-serve via email link, but the community forum shows editing the **account owner's email is NOT self-serve** and is an open feature request — a useful negative data point — [Manage your account](https://docs.retellai.com/accounts/account), [Edit Account Login Information (feature request)](https://community.retellai.com/t/ability-to-edit-account-owner-login-information/347)
3. **Vapi** — voice-AI build platform; email change requires a confirmation link to the new address per community reports, but is inconsistently available and sometimes routed to support instead — [Change Email address for existing account](https://vapi.ai/community/m/1385574268825501717), [How can I change the email for my account?](https://vapi.ai/community/m/1262316118861676544)
4. **Synthflow AI** — no-code voice-agent platform; documents account credentials as "sensitive personal information" it protects, recommends MFA, but publishes no detailed self-service change-email flow — [Trust Vault](https://security.synthflow.ai/), [Account Basics FAQ](https://support.synthflow.ai/hc/en-us/articles/16447735443484-Synthflow-Account-Basics-FAQ)
5. **Bland AI** — enterprise conversational voice-agent console; standard email+password with 2FA available, self-serve password recovery via email link — [Bland Console](https://app.bland.ai/)
6. **PolyAI** — enterprise voice-AI; SOC 2 Type II + ISO 27001 certified, GDPR/HIPAA-aligned, but publishes no consumer-facing change-email/change-password documentation (an enterprise-support-mediated model) — [Compliance](https://docs.poly.ai/legal/compliance)
7. **Goodcall** — AI phone-agent for local/SMB businesses; password change is **not self-serve** — the documented path is "Forgot Password" or a support ticket, a clear low bar for the SMB tier of this competitor set — [How to change my Goodcall password](https://help.goodcall.com/en/articles/8007556-how-to-change-my-goodcall-password)
8. **Smith.ai** — human+AI virtual receptionist; email/contact info is edited via a "User settings → Update information" panel, with the caveat that the listed email doubles as the login identifier; password changes flow through "Forgot password" — [Using and Accessing the Smith.ai Call Dashboard](https://docs.smith.ai/article/n40myw0flr-using-and-accessing-the-smith-ai-call-dashboard), [How to Update Your Contact Information](https://docs.smith.ai/article/4y5sdhuzrr-how-to-update-your-contact-information-on-your-smith-ai-chat-dashboard)
9. **Ruby Receptionists** — human virtual-receptionist portal; password/account management exists in the "Ruby 2.0 Portal" but detailed self-service change-email documentation is not published — [Ruby's Online Portal](https://rubyhelpcenter.helpjuice.com/en_US/apponline-portal/rubys-online-portal)
10. **Rosie** — AI answering service for home-services/SMB; mobile app manages call handling but publishes no detailed change-password/change-email documentation — [Rosie Mobile App](https://heyrosie.com/features/mobile-app)

**Grounding reference (not a competitor, cited because it is the canonical authority for this exact flow):** OWASP — *Changing A User's Registered Email Address For An Account* — [owasp.org](https://owasp.org/www-community/pages/controls/Changing_Registered_Email_Address_For_An_Account). This is the single most load-bearing source for this catalog: it is the only source among everything surveyed that specifies the full dual-confirmation, self-invalidating-token, re-authentication design the product's own bullets already imply.

**Notable finding up front:** self-service, token-confirmed email change is thin-to-absent across the literal AI-receptionist/voice-agent competitor set — Goodcall pushes password changes to a support ticket, Retell AI's own community treats owner-email editing as an unshipped feature request, and most of the rest simply don't publish the flow. The product's own documented bullet ("Confirms the new address by emailed token before it replaces the old one") is therefore not "what Bland AI does" so much as it is the OWASP-canonical, security-conscious design that the *better-resourced* platforms in this set (Dialpad, PolyAI) gesture at but don't fully document, and that the SMB tier skips entirely. Building it properly is a differentiator against the SMB competitors and table-stakes against the enterprise ones.

## Feature catalog (this sub-module only)

### Change Password
- **Current-password re-entry required before a new password is accepted** — the change-password form has an `old_password` field checked via `user.check_password()`, separate from the new-password fields · seen in: Dialpad (explicit "Enter your current password" step), OWASP (mandatory password re-entry for a sensitive change when MFA is absent) · priority: **REQUIRED** (explicitly the documented bullet; also the standard control against a hijacked-but-not-yet-logged-out session) · model: reuses `accounts.User.password` (existing column, no schema change) · realtime: N/A — web/session flow, not call runtime · tool-surface: pure backend/UI — new `ChangePasswordForm(forms.Form)` in `apps/accounts/forms/Auth.py`, modelled on the existing `SetNewPasswordForm` but adding the current-password check; no LLM tool · buildable now.
- **Django password validators enforced on the new password** — the exact `password_validation.validate_password(new, user)` call already used by `SetNewPasswordForm.clean()`, including the similarity-to-user-attributes check · seen in: universal SaaS/NIST 800-63B baseline; already implemented once in this codebase · priority: **REQUIRED** (documented: "enforces Django's validators") · model: none — reuses `accounts.User` instance passed into the validator · realtime: N/A · tool-surface: pure backend, reuse of existing pattern · buildable now.
- **Session re-authenticated in place — `update_session_auth_hash(request, user)`** — because changing `password` invalidates Django's session auth hash, the authenticated change-password view must call this immediately after `set_password()`+`save()` or the user is logged out mid-flow · seen in: Django framework guidance itself; universal pattern across every SaaS reviewed (Dialpad, GitHub, Google all keep the *current* session alive after an in-app password change) · priority: **REQUIRED** (this is literally the documented bullet: "re-authenticates the session so the user is not logged out") · model: none — Django session framework · realtime: N/A · tool-surface: pure backend, one function call · buildable now, zero dependency.
- **Rate-limiting the change-password endpoint itself** — an authenticated attacker (stolen session/CSRF) probing the current-password field is a live account-takeover attempt, not a hypothetical; throttle by `(user_id)` and by client IP using the same cache-backed mechanism as 0.1's login throttling · seen in: NIST SP 800-63B (rate-limit any authentication-adjacent endpoint, not only login), extends 0.1's own `throttling.py` design · priority: **REQUIRED** · model: none — reuses `apps.accounts.throttling.build_keys()`/`is_throttled()`/`register_failure()` with a new `'change_password'` scope string, zero changes to the throttling module itself · realtime: N/A · tool-surface: pure backend · buildable now, reuses existing infra entirely.
- **"Sign out of other sessions" on password change** — invalidates every session except the current one, so a password change actually evicts an attacker who is mid-session elsewhere · seen in: Dialpad ("Force logout" as an admin-side analogue), GitHub/Google self-serve session revocation on password change · priority: differentiator — present only on the enterprise-postured platforms, absent from every SMB tool surveyed (Goodcall doesn't even have self-serve password change) · model: needs session-to-user tracking beyond `accounts.User`/`accounts.UserLocation` — same gap 0.1 already identified and deferred for "Force logout / remote session termination" · **flag, do not build** · realtime: N/A · tool-surface: none · **deferred**, same reasoning as 0.1's identical deferred item — do not re-solve here.

### Change Email
- **Confirm-before-replace via a signed, single-use, short-TTL token emailed to the new address** — the address is written to `accounts.User.email` only after the link is clicked, never at request time · seen in: OWASP's canonical email-change control (confirmation to new address is one of two required legs); implicitly the product's own bullet · priority: **REQUIRED** · model: **no new table** — the pending change is carried entirely inside a `django.core.signing.dumps({'user_id': ..., 'new_email': ..., 'old_email': ...})` payload, verified with `loads(token, max_age=settings.EMAIL_CHANGE_TOKEN_MAX_AGE)` (already scaffolded at 3600s). **Self-invalidation mechanism:** the confirm view compares `user.email == payload['old_email']` before applying — if the email already changed by any route (this token was already used, or a newer request superseded it, or an admin changed it via 0.3's later user-edit surface), the captured `old_email` no longer matches and the token is rejected as stale rather than reapplying a now-wrong change · realtime: N/A · tool-surface: pure backend — new `email_change_confirm_view(request, token)`, GET-only (the link itself is the action, matching the existing `password_reset_confirm_view` shape) · buildable now, zero external dependency beyond the already-configured `EMAIL_BACKEND`.
- **`(tenant, email)` uniqueness re-validated AT CONFIRMATION time, not only at request time** — the address could be claimed by a second user inside the token's TTL window, so the confirm view must re-check `User.objects.filter(tenant=user.tenant, email__iexact=new_email).exclude(pk=user.pk).exists()` immediately before saving, and fail gracefully (not a 500, not an `IntegrityError` bubbling up) if the race lost · seen in: OWASP (re-validate at each step, never trust the state from request time); a direct consequence of the ERD's own `UniqueConstraint(fields=['tenant', 'email'])` · priority: **REQUIRED** · model: reuses `accounts.User`'s existing constraint (verified in `models/User.py` `Meta.constraints`) · realtime: N/A · tool-surface: pure backend · buildable now.
- **Current-password re-entry required to REQUEST an email change** — mirrors the same sensitive-action control as password change; the "request email change" form takes `password` + `new_email`, checked via `user.check_password()` before a token is even issued · seen in: OWASP (password re-entry mandatory for a sensitive account mutation absent MFA), Dialpad's general identity-verification posture for account changes · priority: **REQUIRED** · model: none — reuses `accounts.User.password` · realtime: N/A · tool-surface: pure backend — new `ChangeEmailRequestForm(forms.Form)` · buildable now.
- **Rate-limiting the change-email-request endpoint** — same reasoning as the change-password throttle: an authenticated attacker probing the current-password field on THIS form is exactly as dangerous as on the password form · seen in: NIST 800-63B, extends 0.1's throttling design · priority: **REQUIRED** · model: none — new `'change_email'` throttle scope on the existing `throttling.py` · realtime: N/A · tool-surface: pure backend · buildable now.
- **Masked "current email" shown on the request form** so the user confirms which address they're replacing before submitting · seen in: general SaaS UX baseline (Smith.ai's own "Update information" panel shows the current value inline) · priority: table-stakes · model: none — read-only display of `request.user.email` · realtime: N/A · tool-surface: pure UI · buildable now.
- **New-address confirmation link explicitly states an expiry window** ("this link expires in 1 hour") so a stale click gets a clear, friendly failure rather than a generic error · seen in: OWASP, Vaadata (email-change best practices) · priority: table-stakes · model: none — derived from `EMAIL_CHANGE_TOKEN_MAX_AGE`, already a settings value · realtime: N/A · tool-surface: pure UI copy · buildable now.

### Credential Change Notice
- **Generalize the existing `_send_password_changed_email` / `CHANGED_EMAIL_SUBJECT` / `CHANGED_EMAIL_BODY` helper into one shared "credential change notice" utility with two call sites** — (1) password changed (already wired from 0.1's unauthenticated `password_reset_confirm_view`; 0.2 adds the authenticated `change_password_view` as a second caller) and (2) email changed (new — fires in `email_change_confirm_view`, addressed to the **OLD** address, per the documented bullet: "Emails the previous address whenever the password or email changes") · seen in: OWASP's "notify the current/previous address" control; this codebase's own 0.1 precedent and its explicit forward-reference comment · priority: **REQUIRED** · model: none — pure refactor of existing `send_mail(...)` call pattern into a parameterized helper (e.g. `_send_credential_change_notice(user, kind, detail=None)` living in `apps/accounts/views/_helpers.py` since it's now used by more than one entity module, per the backend-structure rule that shared-across-sub-modules helpers belong there) · realtime: N/A · tool-surface: pure backend · buildable now — this is a refactor of code that already exists, not new integration surface.
- **Notice states plainly WHAT changed** ("your password was changed" vs. "your email was changed to X") rather than one generic message, so the recipient can judge legitimacy at a glance · seen in: OWASP, Vaadata (be explicit about the nature of the change) · priority: table-stakes · model: none — two body templates (or one parameterized template) alongside the existing `CHANGED_EMAIL_SUBJECT`/`CHANGED_EMAIL_BODY` constants · realtime: N/A · tool-surface: pure backend copy · buildable now.
- **Notice delivery never blocks or fails the credential change itself** — `fail_silently=True` + `logger.exception(...)` on send failure, exactly the pattern already in `_send_password_changed_email` · seen in: this codebase's own existing 0.1 code · priority: **REQUIRED** (a notification outage must never surface as a broken password/email change, and must never silently swallow the fact that a takeover tripwire failed to fire — hence the `logger.exception`, which is the operational side of this requirement) · model: none · realtime: N/A · tool-surface: pure backend · buildable now, literally already-proven code.
- **Notice includes an explicit "if this wasn't you" call-to-action** pointing at the administrator, not a bare confirmation · seen in: OWASP, and the existing `CHANGED_EMAIL_BODY` copy in `views/Auth.py` already does this — 0.2 only needs to mirror the same phrasing into the email-changed variant · priority: **REQUIRED** (documented as "the account-takeover tripwire") · model: none · realtime: N/A · tool-surface: pure backend copy · buildable now.

### Beyond the bullets
- **Dual notification at REQUEST time, not just at apply time** — OWASP's stricter recommendation is to also alert the *old* address the moment an email change is *requested* (before confirmation), not only once it's applied. The product's own documented bullet only mandates the after-the-fact notice ("whenever the password or email changes" — i.e., once the change actually takes effect), so this is an **optional enhancement beyond what's required**, not a gap · seen in: OWASP · priority: differentiator (would strengthen the tripwire further, but is not required to satisfy the documented feature) · model: none — same helper, called from `change_email_request_view` too · realtime: N/A · tool-surface: pure backend · **buildable now if wanted, but not required by the documented bullet** — flag as an optional strengthening for `todo` to accept or decline, do not treat as mandatory scope.
- **Pending-change visibility has a real constraint under the zero-model rule** — because there is no stored row for "an email change is pending," the account page cannot show a persistent "you have a pending change to newemail@x.com, resend?" banner across page loads purely from the database. The two compliant options are: (a) a one-time flash message immediately after the request POST ("check newemail@x.com to confirm — link expires in 1 hour"), with no further persistent state, or (b) echo the pending new-email into `request.session` (session data, not a DB column — ephemeral, per-device, not a new model) purely so the account page can render "pending: newemail@x.com" for THAT browser session until it's confirmed, expires, or is superseded. Recommend (a) as the minimum; (b) as an optional UX nicety `todo` may include without violating the zero-model constraint · priority: table-stakes (design clarification, not a feature to skip) · model: none · realtime: N/A · tool-surface: pure UI/session · buildable now.
- **Security/audit log of credential-change events** ("password changed on 2026-07-01 from IP X") visible to the user · seen in: Dialpad's Account Security page, general enterprise SaaS pattern · priority: differentiator · model: would need a new table — no home in the eleven-model set; this is the same gap 0.1 already identified and deferred as "Login audit trail," now extended to credential-change events specifically · **flag, do not build** · realtime: N/A · tool-surface: none · **deferred**, same reasoning as 0.1.
- **CAPTCHA / step-up MFA challenge before a sensitive credential change** · seen in: general industry practice, not specifically published by any surveyed competitor for this exact flow · priority: differentiator · model: none needed for CAPTCHA (third-party dependency); MFA needs new `User` fields at minimum — same reasoning 0.1 already deferred for TOTP MFA · **deferred**, external dependency / same gap as 0.1.

## Compliance & provider constraints

- **REQUIRED — Credential Change Notice is the product's own account-takeover tripwire**, explicitly named as such in the documented bullet. It is never optional and never deferred: every password change and every applied email change fires the notice to the address that was replaced (old email on an email change; the account's email on a password change), regardless of which of the two entry points (0.1's unauthenticated reset-confirm, or 0.2's authenticated change) triggered it.
- **REQUIRED — password re-entry (re-authentication) before both sensitive mutations**, per OWASP's guidance for changing a registered email/password absent MFA. Neither the change-password nor the change-email-request form may proceed without checking the CURRENT password.
- **REQUIRED — signed, single-use, short-TTL token for the email-change confirmation link**, self-invalidating by embedding the pre-change email and re-checking it at confirm time — the same design principle Django's own `PasswordResetTokenGenerator` already uses for 0.1's reset link (embedding state that a successful change invalidates), applied here via `django.core.signing` since there is no persisted row to hash against.
- **REQUIRED — rate-limiting on both authenticated endpoints** (change-password, change-email-request), because a valid session with a wrong "current password" is a live account-takeover probe, not a hypothetical one; this extends the identical REQUIRED throttling obligation 0.1 already established for login and password-reset-request.
- **PII-logging discipline extends to this sub-module.** Never log the old or new email address, the current or new password, or the raw signing token at INFO — only `logger.info('Password changed for user_id=%s tenant_id=%s', ...)` / `'Email change requested/confirmed for user_id=%s tenant_id=%s'`, mirroring 0.1's existing `logger.info('Login succeeded for user_id=%s tenant_id=%s', ...)` shape.
- **This sub-module has no telephony/recording/HIPAA/two-party-consent surface.** It never touches `calls.CallSession` and appends nothing to `.usage`. GDPR applies only in the generic sense that an email address is personal data and this sub-module IS the self-service rectification (Article 16) path for it — no additional obligation beyond what 0.1 already noted for `email`/`primary_phone`.
- **No provider cost line.** The only external dependency is outbound email (the confirmation link and the two notice variants), already routed through the same configurable `EMAIL_BACKEND` (console in dev) 0.1 uses — no new integration, no Twilio/STT/TTS/LLM involvement, nothing appended to any `calls.CallSession.usage`.
- **Cache-backend correctness carries over from 0.1's note** — the new `change_password`/`change_email` throttle scopes share `throttling.py`'s existing `LocMemCache`-in-dev / Redis-in-production caveat; not a new operational concern, just an extension of the existing one.

## Recommended build scope (this pass)

**0.2 introduces ZERO new models and ZERO migrations — confirmed correct by research, not merely mandated.** Every researched feature above is satisfied by mutating the two existing columns on the already-migrated `accounts.User` (`email`, `password`) through new views, plus one new use of `django.core.signing` for transient, self-invalidating token state, plus two new scopes on the already-generic `throttling.py`. Nothing researched needs a persisted "pending change" row — the token TTL (`EMAIL_CHANGE_TOKEN_MAX_AGE`, already in settings) and the embedded old-email self-check together do everything a `PendingEmailChange` table would have done, without adding a twelfth model.

What this pass touches, concretely, for `todo` to plan against:

- **`accounts.User`** (existing, verified, no schema change) — `email` and `password` columns are read/written by the new views; no new field, no new migration. FKs unaffected.
- **`apps/accounts/forms/Auth.py`** (existing file, extended) — add `ChangePasswordForm` (old password + new password × 2, reusing `SetNewPasswordForm`'s validator pattern) and `ChangeEmailRequestForm` (password + new email, reusing `style_widgets`). Both plain `forms.Form`, no tenant-scoped FK, matching the file's existing convention.
- **`apps/accounts/views/Auth.py`** (existing file, extended) — add `change_password_view`, `change_email_request_view`, `email_change_confirm_view`. Generalize `_send_password_changed_email`/`CHANGED_EMAIL_SUBJECT`/`CHANGED_EMAIL_BODY` into a shared credential-change-notice helper (relocate to `apps/accounts/views/_helpers.py` since it now serves more than one entity module, per the backend package-structure rule).
- **`apps/accounts/throttling.py`** (existing file, unchanged code — new call-site scopes only) — `change_password` and `change_email` scopes added at the call sites in the two new views; the module itself needs no edits since `build_keys()` already takes an arbitrary scope string.
- **`apps/accounts/urls.py`** (existing flat module, extended) — three new literal routes appended after the 0.1 block and before any later `crud()` call: `accounts:change_password`, `accounts:change_email` (request), and the confirm link (e.g. `accounts:email_change_confirm`, taking a `<token>` path segment — a literal-prefixed route so it cannot collide with a future `<int:pk>` route per the file's own ordering rule). The two url names `change_password`/`change_email` **must** match what `context_processors.py`'s `OPTIONAL_CHROME_URLS` already expects.
- **First use of `django.core.signing`** in this codebase — `dumps()`/`loads(max_age=...)` for the pending-email-change payload; no new settings needed beyond the already-present `EMAIL_CHANGE_TOKEN_MAX_AGE`.
- **Templates** (new, under the flat foundation-app template rule — Modules 0–1 have no sub-module folder level): `templates/accounts/change_password.html`, `templates/accounts/change_email.html`, `templates/accounts/email_change_confirm.html` (a simple "confirmed" / "link invalid or expired" result page, styled like `password_reset_confirm.html`'s `valid`/invalid branch).

No third or fourth model is justified by anything researched above — every candidate that would have needed one (session-to-user tracking for "sign out elsewhere," an audit-log table, MFA secrets/recovery codes) is the same gap 0.1 already identified and deferred, and none of it is required to satisfy the three documented 0.2 bullets.

## Out of scope for this product (outside the seven capabilities)

- **Billing/notification-preference email management** (Smith.ai's "billing email"/"notification settings" panel) — billing is not one of the seven capabilities.
- **Government-ID identity verification for fraud review** (Dialpad's ID-upload flow for flagged accounts) — this product has no fraud-review team surface; out of scope entirely.
- **Third-party OAuth account linking** (Google/Microsoft sign-in delegation, as Dialpad supports) — 0.1 already established customer-id + email/username + password as the sole login mechanism; OAuth linking is a login-mechanism change owned by 0.1 if ever pursued, not a credential-*change* concern, and is not in scope for either sub-module today.
- **Agency/reseller subaccount permission management** (Synthflow's per-subaccount permission tabs) — this product has no reseller/partner concept; a tenant is a single business.

- **Security/audit log of credential-change events** — needs a new table with no home in the eleven-model set; identical gap to 0.1's deferred "Login audit trail," now simply extended to cover change-password/change-email events too.
- **TOTP MFA / step-up re-authentication before a sensitive change** — would start as extra fields on `accounts.User` rather than a new table (as 0.1 already noted), but is not required to satisfy any of 0.2's three documented bullets; deferred until a security-hardening pass is explicitly scoped.
- **Dual old-address notification at REQUEST time** (OWASP's stricter recommendation, vs. this product's documented apply-time-only notice) — buildable later by adding one more call site to the same shared helper; not required now.
- **CAPTCHA after repeated failed "current password" entries** — third-party dependency, not required for the documented bullets.
- **SSO/OIDC enterprise sign-in and its interaction with local credential management** (e.g., disabling password change entirely for an SSO-linked user) — 0.1 already deferred SSO itself (needs a twelfth table for per-tenant IdP config); 0.2 has nothing to build here until SSO exists.
