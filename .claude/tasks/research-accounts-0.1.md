**Notable finding up front:** none of the ten inbound-AI-receptionist/booking competitors surveyed put a tenant/organization identifier *on the login form itself* the way this product's "Customer-Scoped Login" bullet specifies. They authenticate a single global email identity first (Retell: email+password or "Sign in with Google"; Vapi: JWT/org-scoped API keys but a global dashboard login; Bland: email+password or enterprise SSO) and resolve org/workspace membership *after* authentication, with a post-login org switcher for the rare multi-org user. The **customer-id-first, tenant-resolved-before-authentication** pattern this product specifies is architecturally necessary here for a different reason ‚Äî `(tenant, email)` is unique but `email` alone is not, per the ERD ‚Äî and it matches the enterprise B2B "organization-scoped login" pattern documented by Auth0/WorkOS, not the AI-receptionist competitor set. This is a correct, deliberate design choice; it just isn't "what Bland AI does," and the catalog below reflects that honestly rather than force-fitting a citation.

## Feature catalog (this sub-module only)

### Customer-Scoped Login
- **Tenant resolution before credential check** ‚Äî `customer_id` looks up `Tenant` first; the email-or-username + password check then runs scoped to that tenant only, so the same email can exist in two businesses without collision ¬∑ seen in: Auth0 Organizations, WorkOS (organization-scoped login pattern) ‚Äî not literally present in the AI-receptionist competitor set (see note above) ¬∑ priority: **table-stakes for multi-tenant B2B SaaS**, architecturally required here because `(tenant, email)` is unique and `email` alone is not ¬∑ model: reuses `tenants.Tenant` (verified, `customer_id` field already unique) + new `accounts.User` (tenant-scoped) ¬∑ realtime: N/A ‚Äî web/session flow, not call runtime ¬∑ tool-surface: pure UI/backend view (custom `AUTHENTICATION_BACKENDS` entry), no LLM tool ¬∑ buildable now (no external dependency).
- **Inactive-tenant gate at login** ‚Äî a tenant marked inactive is refused at the login form with a clear message, never mid-session ¬∑ seen in: this product's own documented behavior (module-1 bullet "Tenant Activation State" explicitly calls out "blocked at login rather than mid-call") ‚Äî general SaaS practice (Dialpad/Bland suspend org access centrally) ¬∑ priority: table-stakes ¬∑ model: reuses `tenants.Tenant.is_active` (verified field, already exists) ‚Äî **the toggle UI belongs to sibling 1.1**; 0.1 only reads it ¬∑ realtime: N/A ¬∑ tool-surface: pure UI/backend check ¬∑ buildable now.
- **Email-or-username interchangeable identifier** ‚Äî a user may sign in with either, matched case-insensitively, within the resolved tenant ¬∑ seen in: Retell AI (email or Google), most SaaS login forms generally ¬∑ priority: table-stakes ¬∑ model: reuses `accounts.User.email` / `.username` (both already fielded in ERD ¬ß3.1, `username` nullable/blank) ¬∑ realtime: N/A ¬∑ tool-surface: pure backend (custom auth backend `get_by_natural_key`-equivalent logic) ¬∑ buildable now.
- **SSO / SAML / OIDC enterprise sign-in** ‚Äî sign in via the tenant's own identity provider (Okta, Azure AD, Google Workspace) instead of a local password ¬∑ seen in: Bland AI (OIDC/SAML, gated to enterprise contracts), Vapi (JWT/enterprise sign-on), Synthflow (SOC2/ISO-aligned enterprise tier), Retell AI (Azure AD SSO), Dialpad (SAML via Azure/OneLogin/Google Workspace) ¬∑ priority: **differentiator** ‚Äî present only on the enterprise tiers of the larger platforms, absent from the SMB-facing tools (Smith.ai, Ruby, Rosie, Goodcall have no published SSO) ¬∑ model: would need a **twelfth table** (per-tenant IdP connection: issuer, client id/secret, protocol) ‚Äî **do not build**; flag only ¬∑ realtime: N/A ¬∑ tool-surface: none ¬∑ **integration/later** ‚Äî external IdP dependency, out of this pass; see Deferred.

### Logout & Session Expiry
- **Explicit logout, session + active-location clear** ‚Äî ends the Django session and clears the active-location key from session data in the same request ¬∑ seen in: universal SaaS pattern (Dialpad "Force logout" is the admin-side analogue) ¬∑ priority: table-stakes ¬∑ model: no new model ‚Äî Django's session framework + `request.session.pop('active_location_id', ...)`; the *field* the switcher reads is `accounts.UserLocation`'s existence, not stored on session beyond the id ¬∑ realtime: N/A ¬∑ tool-surface: pure backend view ¬∑ buildable now.
- **Inactivity timeout, per user** ‚Äî after N idle minutes the session is invalidated server-side on the next request, forcing re-login ¬∑ seen in: Dialpad (admin session controls / force logout), general enterprise-SaaS session-idle policy referenced by Bland/Synthflow's SOC2-aligned posture ¬∑ priority: **common** among the enterprise-tier competitors, absent/unpublished among the SMB tools ¬∑ model: reuses `accounts.User.inactivity_timeout` (verified field in ERD ¬ß3.1, minutes) with `DEFAULT_INACTIVITY_TIMEOUT_MINUTES` as the fallback (already in settings.py) ‚Äî last-activity timestamp lives in session data (`request.session['last_activity']`), **no new table** ¬∑ realtime: N/A ¬∑ tool-surface: pure backend (`SessionPolicyMiddleware`, already named in settings.py) ¬∑ buildable now.
- **Absolute session lifetime ceiling** ‚Äî a session expires after a fixed maximum regardless of activity, independent of the idle timeout ¬∑ seen in: general SaaS/security baseline (NIST session-management guidance) ¬∑ priority: table-stakes ¬∑ model: none ‚Äî Django's `SESSION_COOKIE_AGE` ¬∑ realtime: N/A ¬∑ tool-surface: none ¬∑ buildable now.
- **Force logout / remote session termination** ‚Äî an admin can end another user's active sessions immediately (e.g. after an offboarding or a suspected compromise) ¬∑ seen in: Dialpad ("Force logout" ends all sessions on a linked device) ¬∑ priority: differentiator ¬∑ model: needs session-to-user tracking beyond the two fixed models (Django's DB session table has no user index by default) ‚Äî **flag as needing schema growth beyond the ERD's fixed fields; do not build now** ¬∑ realtime: N/A ¬∑ tool-surface: none ¬∑ **deferred**.

### Forgot & Reset Password
- **Non-enumerating reset request** ‚Äî the "forgot password" form always responds with the same "if that account exists, a reset link was sent" message, whether or not the email/customer id matched anything ¬∑ seen in: NIST SP 800-63B guidance, universal SaaS best practice (all ten competitors' reset flows follow this shape, e.g. Smith.ai's forgot-password on the call dashboard) ¬∑ priority: **REQUIRED** (account-existence disclosure) ¬∑ model: none ¬∑ realtime: N/A ¬∑ tool-surface: pure backend view ¬∑ buildable now.
- **Tenant-disambiguated reset** ‚Äî because `email` alone is not globally unique (only `(tenant, email)` is), the reset form must either also collect `customer_id` (mirroring the login form) or, given a bare email that matches multiple tenants, issue one reset link per matching tenant without revealing how many tenants matched ¬∑ seen in: this is a direct consequence of the ERD's own uniqueness constraint, not a competitor pattern ‚Äî none of the surveyed products are multi-tenant-with-shared-email at this depth ¬∑ priority: **REQUIRED** (architecturally forced by the data model; getting this wrong either enumerates tenants or silently resets the wrong tenant's account) ¬∑ model: reuses `tenants.Tenant` + `accounts.User`, no new table ¬∑ realtime: N/A ¬∑ tool-surface: pure backend view ¬∑ buildable now.
- **Single-use, short-TTL signed token** ‚Äî the reset link encodes a signature that expires after a fixed window and is invalidated by any subsequent use (a successful reset changes the password hash the token is derived from, which naturally invalidates it) ¬∑ seen in: universal pattern; Django's built-in `PasswordResetTokenGenerator` implements exactly this ¬∑ priority: table-stakes ¬∑ model: none ‚Äî Django built-in, driven by the already-present `PASSWORD_RESET_TIMEOUT` setting (3600s) ¬∑ realtime: N/A ¬∑ tool-surface: pure backend view ¬∑ buildable now, zero external dependency beyond the already-configured email backend.
- **Reset-request throttling** ‚Äî the forgot-password endpoint is itself rate-limited per email/IP so it cannot be used to mail-bomb an inbox or probe for valid customer ids ¬∑ seen in: NIST 800-63B (applies the same throttling principle to any authentication-adjacent endpoint) ¬∑ priority: **REQUIRED** ¬∑ model: none ‚Äî same cache-based mechanism as login throttling ¬∑ realtime: N/A ¬∑ tool-surface: pure backend view ¬∑ buildable now.
- **Post-reset confirmation email** ‚Äî after a successful reset, an email is sent confirming the change, as the tripwire for a reset the account owner didn't request ¬∑ seen in: general SaaS security baseline (Dialpad's account-security emails, NIST's "notify via out-of-band channel" guidance) ¬∑ priority: common ¬∑ model: none ‚Äî reuses the email backend ¬∑ **note:** this is the *unauthenticated forgot/reset* path's own confirmation; keep it distinct from 0.2's "Credential Change Notice" (which fires on the *authenticated* change-password/change-email flows) ‚Äî the two should likely share one notification helper, but that helper is 0.2's to build; 0.1 only needs the call site ¬∑ realtime: N/A ¬∑ tool-surface: pure backend view ¬∑ buildable now (coordinate helper ownership with 0.2 during `todo`).

### Failed-Attempt Throttling
- **Per-account AND per-IP throttling, cache-backed** ‚Äî failed attempts are counted both against the specific `(customer_id, identifier)` pair and against the source IP, in a rolling window, with no new table ¬∑ seen in: NIST SP 800-63B / AC-7 (recommends IP-based throttling alongside account-based), Dialpad's account-security posture ¬∑ priority: **REQUIRED** ¬∑ model: none ‚Äî Django cache (`LOGIN_ATTEMPT_LIMIT` / `LOGIN_ATTEMPT_WINDOW_SECONDS`, already defined in settings.py) ¬∑ realtime: N/A ¬∑ tool-surface: pure backend (auth backend / login view) ¬∑ buildable now, **operational caveat:** `CACHES['default']` is `LocMemCache` in dev, which is per-process ‚Äî a multi-worker deployment needs a shared cache (Redis, already provisioned for Channels) for the throttle counters to be correct across workers; flag for the deployment pass, not blocking for dev/tests.
- **Uniform failure message (account-existence non-disclosure)** ‚Äî "invalid customer id, username/email or password" regardless of which part was wrong, never "no such customer id" or "no such user" ¬∑ seen in: NIST 800-63B, universal SaaS practice ¬∑ priority: **REQUIRED** ¬∑ model: none ¬∑ realtime: N/A ¬∑ tool-surface: pure backend view ¬∑ buildable now.
- **Progressive backoff / lockout with high ceiling** ‚Äî delay grows after each failure rather than an immediate hard lockout at a low count; NIST's own ceiling is 100 attempts with throttling, though most production systems (this product's `LOGIN_ATTEMPT_LIMIT=5`) lock out earlier ¬∑ seen in: NIST SP 800-63B ¬ß5.2.2 ¬∑ priority: table-stakes (the specific threshold is a config value, not a competitor feature) ¬∑ model: none ‚Äî cache counter + window, already scaffolded in settings.py ¬∑ realtime: N/A ¬∑ tool-surface: pure backend view ¬∑ buildable now.
- **Lockout notification / CAPTCHA after repeated failures** ‚Äî an out-of-band notice on lockout, or a CAPTCHA challenge after N failures, as an added friction layer ¬∑ seen in: NIST guidance (notify via out-of-band channel), general industry practice (not specifically published by any of the ten competitors) ¬∑ priority: differentiator ¬∑ model: none, but CAPTCHA needs a third-party verification call (reCAPTCHA/hCaptcha) ¬∑ realtime: N/A ¬∑ tool-surface: none ¬∑ **deferred** ‚Äî external dependency, not required to satisfy the documented bullet.

## Compliance & provider constraints

- **Account-existence non-disclosure is REQUIRED, not table-stakes-only** ‚Äî both the login failure message and the forgot-password response must never reveal whether a `customer_id`, username/email, or account exists. This is the single compliance-adjacent obligation this sub-module carries (NIST SP 800-63B, OWASP ASVS V2.2/V2.3 authentication-failure guidance).
- **PII-logging discipline extends here even though this sub-module never touches a call.** The project-wide rule against logging raw PII applies to login/reset flows too: never log the attempted password, never log the full email in a failed-attempt log line beyond what's needed for the throttle key, and never log the raw reset token (only that a reset was requested/completed).
- **This sub-module has no telephony/recording/HIPAA/GDPR call-retention surface.** It does not touch `calls.CallSession`, does not append to `.usage`, and has no consent-basis or two-party-consent obligation ‚Äî those apply starting at Module 3 (Call Runtime) and Module 5 (Call Logs). GDPR still applies in the generic sense that `email`/`primary_phone` are personal data subject to standard access/erasure rights, but that is a Module 0.3 (profile) concern, not 0.1's.
- **No provider cost line.** Nothing in 0.1 calls Twilio, STT, TTS or an LLM, so nothing here appends to `calls.CallSession.usage`. The only external dependency is outbound email (reset link, and later the 0.2 credential-change notice) ‚Äî already routed through Django's configurable `EMAIL_BACKEND` (console in dev), no new integration needed for this pass.
- **Cache-backend correctness is an operational constraint, not a compliance one** ‚Äî see the throttling note above (`LocMemCache` is per-process; production needs a shared cache).

## Recommended build scope (this pass)

This is effectively a **foundation sub-module**: it defines the two ERD-fixed models for the `accounts` app, but it ships **authentication-flow pages** (login, logout, forgot-password, reset-password), not list/detail/edit/delete CRUD for either model ‚Äî that CRUD belongs to sibling sub-modules named below. At most two models, exactly as instructed:

- **`accounts.User`** (`AUTH_USER_MODEL`) ‚Äî tenant-scoped, not location-scoped ‚Äî fields per ERD ¬ß3.1: `tenant` FK, `email`, `username` (nullable/blank), `first_name`/`last_name`/`full_name`, `primary_phone`, `tier`, `status`, `password` (Django hasher), `last_login_at`, `is_provider`, `provider_hours` (JSON), `inactivity_timeout`. Unique `(tenant, email)`; unique `(tenant, username)` where not null. Justified by: **Customer-Scoped Login** (tenant FK + email/username fields + the custom `CustomerScopedBackend`), **Logout & Session Expiry** (`inactivity_timeout` drives `SessionPolicyMiddleware`), **Forgot & Reset Password** (`password` field is what Django's built-in token generator derives from; `email` is the delivery address), **Failed-Attempt Throttling** (`status` field's `suspended` value is the account-level lock state the throttle can escalate to). FK: `tenants.Tenant` (verified). 0.1 ships the login/session/reset flows against this model; it does **not** ship the "create/edit user" or "own profile" pages (0.3) or the tier/status management UI (0.3).
- **`accounts.UserLocation`** ‚Äî tenant-scoped ‚Äî `tenant` FK, `user` FK (`AUTH_USER_MODEL`), `location` FK; unique `(user, location)`. Justified by: it is the table the active-location switcher (0.4) and every location-scoped queryset's cross-location IDOR check will read, and it must exist as soon as `AUTH_USER_MODEL`'s app is first migrated. **0.1 ships zero CRUD and zero UI for this model** ‚Äî no assignment matrix (that's 1.3), no switcher (that's 0.4). Its migration lands in this pass purely so the two accounts-app models are created together; nothing in 0.1's own login/logout/reset/throttle flows queries it.

No third model is justified by 0.1's researched features. Everything else researched above (throttling counters, session timeout, reset tokens) is deliberately cache/session/Django-builtin-backed per the constraint against a twelfth table.
