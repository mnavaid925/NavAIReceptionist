{% comment %} Filter bar for the call log. Four axes plus search, and every one of them compares against a plain string — there is no FK filter on this page, so nothing here needs |stringformat:"d". `status` and `mode` are model choices; `outcome` is a DERIVED axis read out of the `transfer` JSON column, which is why its choices come from the view's own OUTCOME_CHOICES rather than from a model. OUTCOME_CHOICES ships WITHOUT a leading blank pair (the view's list starts at `no_transfer`), so the "Any outcome" option is written out here. The same is true of status and mode. Do not add a blank pair to the view lists to tidy this up: those lists are also what `_apply_outcome_filter` and the two `dict(...)` membership checks validate against, and a '' member would make the empty string a *valid filter value* instead of the "no filter" it has to mean. The date inputs are LOCAL calendar days at the active location. The view turns each into a half-open UTC range via `local_day_bounds_utc` rather than using `started_at__date`, which converts in the wrong timezone and, on MySQL, returns nothing at all. Nothing about that is visible from here — this bar just posts `?from=`/`?to=` and the view does the honest thing with them. Every filter degrades to "no filter" on junk rather than raising, so a hand-edited query string can never 500 the page it sits on. {% endcomment %}
